# 系统设计与高并发面试指南

## 系统设计面试核心框架

系统设计面试没有标准答案，面试官考察的是你的**分析思路**和**权衡取舍能力**。推荐使用以下 6 步框架：

### STEP 1: 明确需求

- **功能性需求**：系统需要做什么？
- **非功能性需求**：QPS、延迟、可用性、一致性要求
- **约束条件**：技术栈、预算、团队规模

例如："设计一个短链接服务"
- 功能：长 URL → 短 URL 转换、重定向、统计访问量
- 非功能：QPS 10万+、可用性 99.9%、全球低延迟
- 存储：10亿+ 记录

### STEP 2: 顶层设计

画一个简单的架构图：客户端 → API 网关 → 业务服务 → 数据层

### STEP 3: 深入组件设计

逐个深入核心组件的设计细节。

### STEP 4: 数据存储设计

根据数据特征选择合适的存储方案。

### STEP 5: 可扩展性与高可用

考虑水平扩展、故障恢复、监控告警。

### STEP 6: 权衡与优化

讨论方案的 trade-off，提出可能的优化方向。

## 高并发架构核心知识

### 1. 缓存架构

**缓存策略选择：**

| 策略 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| Cache Aside | 读多写少 | 实现简单 | 首次请求较慢 |
| Read/Write Through | 数据一致性要求高 | 对应用透明 | 实现复杂 |
| Write Behind | 写密集 | 写性能高 | 有数据丢失风险 |

**缓存常见问题与解决方案：**

- **缓存穿透**：查询不存在的数据 → 布隆过滤器 + 空值缓存
- **缓存击穿**：热点 key 过期 → 互斥锁 + 逻辑过期
- **缓存雪崩**：大量 key 同时过期 → 随机过期时间 + 多级缓存

**多级缓存架构：**

请求 → CDN → Nginx 本地缓存 → Redis 分布式缓存 → 数据库

### 2. 消息队列

**核心作用：** 解耦、异步、削峰

**常见 MQ 对比：**

| 特性 | RabbitMQ | Kafka | RocketMQ |
|------|----------|-------|----------|
| 吞吐量 | 万级 | 百万级 | 十万级 |
| 延迟 | 微秒级 | 毫秒级 | 毫秒级 |
| 消息可靠性 | 高 | 高 | 高 |
| 适用场景 | 业务消息 | 日志/大数据 | 电商/金融 |

**消息可靠性保障：**
- 生产端：确认机制 + 本地消息表
- 存储端：多副本 + 同步刷盘
- 消费端：手动 ACK + 幂等设计

### 3. 数据库优化

**读写分离：** 主库写 → 从库读，通过主从复制同步数据

**分库分表：**
- 垂直分库：按业务模块拆分数据库
- 水平分表：按用户 ID 哈希或范围路由

**索引优化：**
- 遵循最左前缀原则
- 避免在索引列上做函数运算
- 覆盖索引减少回表
- 注意索引失效场景（隐式类型转换、LIKE '%xx'）

### 4. 限流降级

**限流算法：**

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 计数器 | 固定窗口计数 | 简单 | 窗口边界突刺 |
| 滑动窗口 | 细分时间窗口 | 更平滑 | 实现稍复杂 |
| 漏桶 | 固定速率处理 | 流量整形 | 无法应对突发 |
| 令牌桶 | 按固定速率生成令牌 | 允许突发 | 实现较复杂 |

**降级策略：**
- 熔断（Circuit Breaker）：连续失败达到阈值，暂停调用
- 降级：核心功能不可用时提供简化版替代
- 限流：超出的请求直接拒绝或排队

## 常见系统设计面试题

### 设计一个 URL 短链服务

核心方案：自增 ID → Base62 编码 → 分布式 ID（雪花算法）
存储：Redis + MySQL，短码作为 key
优化：布隆过滤器防重复、LRU 缓存热点短链、Bloom Filter 判断存在性

### 设计一个秒杀系统

核心挑战：瞬时高并发 + 防超卖
方案：前端限流 → 网关限流 → Redis 预扣库存 → MQ 异步下单 → 数据库最终一致
关键：Redis Lua 脚本原子扣减库存、MQ 削峰、分布式锁

### 设计一个实时聊天系统

核心方案：WebSocket 长连接 + 心跳保活 + 消息队列 + 分布式存储
消息存储：消息索引（Redis）+ 消息内容（MySQL/MongoDB）
离线消息：用户上线后拉取未读消息
群聊优化：消息扇出（写扩散 vs 读扩散）

### 设计一个新闻信息流

核心：推模式（写扩散）vs 拉模式（读扩散）vs 推拉结合
排序：时间序 + 个性化排序（协同过滤 + 内容推荐）
缓存：Feed 列表缓存 + 热点内容缓存

## CAP 定理与分布式一致性

- **C（一致性）**：所有节点看到相同的数据
- **A（可用性）**：每个请求都能得到响应
- **P（分区容错）**：网络分区时系统仍能工作
- 三者只能同时满足两个，实际系统中 P 是必选项，因此选择 CP 或 AP

**一致性协议：**
- **Paxos**：理论完备但实现复杂
- **Raft**：Paxos 的工程实现，易于理解
- **2PC/3PC**：分布式事务协议

## 微服务设计原则

1. **单一职责**：每个服务只做一件事
2. **服务自治**：独立部署、独立数据库
3. **API 网关**：统一入口，负责路由、限流、认证
4. **服务发现**：Nacos / Consul / Eureka
5. **配置中心**：Apollo / Nacos Config
6. **链路追踪**：SkyWalking / Jaeger
7. **熔断降级**：Sentinel / Hystrix
8. **API 版本管理**：URI 版本 / Header 版本
