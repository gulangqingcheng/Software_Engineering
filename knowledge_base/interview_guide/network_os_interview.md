# 网络协议与操作系统面试题

## 计算机网络

### HTTP/HTTPS

**HTTP 请求方法：**
- GET：获取资源，幂等
- POST：提交数据，非幂等
- PUT：更新资源，幂等
- DELETE：删除资源，幂等
- PATCH：部分更新

**HTTP 状态码：**
- 1xx：信息（100 Continue）
- 2xx：成功（200 OK、201 Created、204 No Content）
- 3xx：重定向（301 永久重定向、302 临时重定向、304 未修改）
- 4xx：客户端错误（400 Bad Request、401 Unauthorized、403 Forbidden、404 Not Found、429 Too Many Requests）
- 5xx：服务端错误（500 Internal Server Error、502 Bad Gateway、503 Service Unavailable）

**HTTPS vs HTTP：**
- HTTPS = HTTP + TLS/SSL 加密
- HTTPS 使用 443 端口，HTTP 使用 80 端口
- HTTPS 需要 CA 证书，数据传输加密
- TLS 握手过程：客户端 Hello → 服务端 Hello + 证书 → 密钥交换 → 加密通信

**HTTP/1.1 vs HTTP/2 vs HTTP/3：**
- HTTP/2：多路复用（一个 TCP 连接并行多个请求）、头部压缩、服务器推送
- HTTP/3：基于 QUIC 协议（UDP），解决队头阻塞问题，0-RTT 建连

### TCP/IP

**三次握手：**
1. 客户端 → SYN → 服务端
2. 服务端 → SYN+ACK → 客户端
3. 客户端 → ACK → 服务端

为什么需要三次？防止已失效的 SYN 请求突然到达服务端，造成资源浪费。

**四次挥手：**
1. 主动方 → FIN → 被动方
2. 被动方 → ACK → 主动方（半关闭状态）
3. 被动方 → FIN → 主动方（数据传输完毕）
4. 主动方 → ACK → 被动方（等待 2MSL 后关闭）

为什么需要四次？TCP 全双工，双方各需要一次 FIN 和 ACK。

**TCP vs UDP：**

| 特性 | TCP | UDP |
|------|-----|-----|
| 连接 | 面向连接 | 无连接 |
| 可靠性 | 可靠（确认+重传） | 不可靠 |
| 有序性 | 有序（序列号） | 不保证 |
| 开销 | 大（头部 20 字节） | 小（头部 8 字节） |
| 场景 | HTTP、文件传输 | DNS、视频流、游戏 |

### DNS 解析过程

1. 浏览器缓存
2. 操作系统缓存
3. 本地 DNS 服务器（递归查询）
4. 根域名服务器 → 顶级域名服务器 → 权威域名服务器（迭代查询）

### Cookie vs Session vs Token

| 特性 | Cookie | Session | Token (JWT) |
|------|--------|---------|-------------|
| 存储位置 | 浏览器 | 服务端 | 客户端 |
| 大小限制 | 4KB | 无限制 | 无限制 |
| 跨域 | 需要配置 | 天然支持 | 天然支持 |
| 扩展性 | 差 | 需要共享存储 | 无状态，易扩展 |
| 安全性 | 较低 | 较高 | 取决于实现 |

## 操作系统

### 进程与线程

**进程 vs 线程：**

| 特性 | 进程 | 线程 |
|------|------|------|
| 定义 | 资源分配的基本单位 | CPU 调度的基本单位 |
| 内存 | 独立地址空间 | 共享进程地址空间 |
| 创建开销 | 大 | 小 |
| 通信 | IPC（管道、消息队列、共享内存） | 共享变量（需要同步） |
| 崩溃影响 | 互不影响 | 一个线程崩溃可能影响整个进程 |

**进程调度算法：**
- FCFS（先来先服务）
- SJF（短作业优先）
- RR（时间片轮转）
- 优先级调度
- 多级反馈队列（Linux 默认使用 CFS）

### 内存管理

**虚拟内存：** 让每个进程拥有独立的地址空间，使用页表映射到物理内存。

**页面置换算法：**
- LRU（最近最少使用）：用双向链表 + 哈希表实现
- LFU（最不经常使用）
- FIFO（先进先出）
- Clock（时钟算法）

**内存碎片：**
- 外部碎片：分段存储产生，用紧凑技术解决
- 内部碎片：分页存储产生，页面大小固定导致

### 死锁

**四个必要条件（Coffman 条件）：**
1. 互斥（Mutual Exclusion）
2. 占有并等待（Hold and Wait）
3. 不可抢占（No Preemption）
4. 循环等待（Circular Wait）

**预防策略：**
- 破坏互斥：使用可重入资源（困难）
- 破坏占有并等待：一次性请求所有资源
- 破坏不可抢占：允许抢占
- 破坏循环等待：按顺序请求资源

**检测与恢复：**
- 银行家算法（预防）
- 死锁检测（资源分配图）
- 恢复：终止进程 / 抢占资源

文本处理：grep、awk、sed、cat、tail、head
网络：netstat、ss、curl、ping、traceroute
